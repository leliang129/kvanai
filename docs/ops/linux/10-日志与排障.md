# 10 日志与排障

## 系统日志

常见日志目录：

- `/var/log/syslog`（Ubuntu）
- `/var/log/messages`（RHEL/CentOS）
- `/var/log/auth.log`（认证相关）
- `/var/log/kern.log`（内核相关）

查看方式：

```bash
sudo tail -n 200 /var/log/syslog
sudo grep -i error /var/log/messages
```

## `journalctl`

systemd 日志统一入口。

```bash
journalctl -xe
journalctl -u nginx -n 100 --no-pager
journalctl --since "2026-03-25 10:00:00" --until "2026-03-25 11:00:00"
journalctl -k -n 50
```

实用参数：

- `-u <service>`：按服务过滤
- `-b`：当前启动周期
- `-f`：实时追踪

## 服务日志分析

典型方法：

1. 先看错误峰值时间段。
2. 再按请求 ID / trace ID 串联上下游。
3. 最后匹配变更记录（发布、配置、依赖升级）。

常用组合：

```bash
journalctl -u myapp --since "30 min ago" | grep -E "ERROR|timeout|panic"
```

## 常见问题排查

### 1) 服务启动失败

- `systemctl status <service>`
- `journalctl -u <service> -n 200`
- 检查配置语法、端口占用、权限与依赖路径。

### 2) 端口不通

- `ss -lntp` 看是否监听。
- `curl -v` 测本机服务。
- 检查防火墙和安全组规则。

### 3) 磁盘满

- `df -h` 看分区。
- `du -sh /var/log/* | sort -h` 找大目录。
- 处理“删除未释放”问题：`lsof | grep deleted`。

## Debug 技巧

- 把问题拆成“网络/权限/资源/配置”四类。
- 每次只变更一个变量，避免混淆因果。
- 保留现场证据（日志片段、系统状态、时间线）。
- 形成复盘模板，沉淀成 Runbook。
